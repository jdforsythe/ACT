## ACT site browser SPA

**Status:** Not started.
**Priority:** Required for v0.2 release.
**Lives at:** `apps/site-browser/` (new), deployed to GitHub Pages at `/browser/` alongside the validator (`/validator/`).

### Why

Right now there's no human-friendly way to inspect what a site is publishing as ACT. A user generating ACT from their site has to `curl /.well-known/act.json | jq` and chase IDs through `act/nodes/<id>.json` files by hand — fine for debugging once or twice, but a real adoption blocker. The validator SPA proves a manifest is *well-formed*; this SPA shows what the tree actually *contains* and lets a user navigate it the way an agent would.

This is also the only credible way for an ACT producer to verify that what their components emit matches what their human-facing site shows. Without it, every adopter is squinting at JSON dumps to reverse-engineer the tree shape.

### What it does

A static SPA. Paste a URL into the box (e.g. `https://your-site.example`) and it:

1. Fetches `<URL>/.well-known/act.json`. Renders the manifest header (site name, conformance level, locales, capabilities, advertised URL templates).
2. Fetches the index from `manifest.index_url`. Renders an outline / sidebar of every node: id, type, title, summary, locale.
3. Lets the user click into any node:
   - For non-leaf nodes (have `children`), shows the children inline and offers a "load subtree" action that fetches `subtree_url_template`.
   - For leaf nodes, fetches the node from `node_url_template`, renders title + summary + each block (prose blocks rendered as markdown; data blocks as syntax-highlighted JSON; component blocks as a labeled card showing the block kind and payload).
4. Surfaces `related[]` cross-links as clickable chips so the user can hop around the graph.
5. Indicates per-node provenance from `metadata.source.adapter`, `metadata.locale`, `metadata.translation_status` (so fallback nodes are visible at a glance).
6. Runs entirely client-side. No proxy, no upload — just CORS-mediated fetches against the user's own site.

### UX requirements

- **Pretty.** This is the user's first hands-on experience with ACT. It should feel polished, not like a debug tool.
- **Intuitive.** Anyone who can read a docs site can navigate this. Tree on the left, node detail on the right.
- **Responsive on errors.** When CORS blocks a fetch, when the manifest is malformed, when a referenced node 404s — show a clear message at the right scope (whole-tree vs. just-this-node), not a stack trace.
- **URL-shareable.** Reflect the currently-viewed URL + node ID in the address bar so a user can share `?site=https://example.com&node=cms/en-US/landing/pricing`.
- **Offline-safe.** If the user pastes a `file://` URL or drops in JSON files directly (drag-and-drop), the SPA validates and renders without any network hop.

### Implementation notes

- Same shape as `apps/validator-web/`: Vite + TypeScript, deployed to Pages.
- Should consume `@act-spec/inspector` (the existing `walk` API) for the actual tree traversal — don't reimplement it. The SPA is a UI on top of the inspector library.
- Reuse `@act-spec/validator` to surface schema / conformance issues inline ("this node is missing `summary` — Standard requires it") so the browser doubles as a live linter while you're building.
- The validator SPA's `apps/validator-web/README.md` is a reasonable template for the README.

### Acceptance

- [ ] Loads any of the v0.1 examples by URL when their dev server is running locally (`http://localhost:3000` for nextjs-marketing, `http://localhost:3001` for docusaurus-docs, etc.) and renders the full tree.
- [ ] Loads a deployed ACT-emitting site over the public internet (CORS permitting).
- [ ] Renders prose blocks as readable markdown, data blocks as JSON, marketing/component blocks as labeled cards.
- [ ] Highlights conformance gaps inline at the node where they occur.
- [ ] Linked from the top-level `README.md` and from each example README under "View this site in the ACT browser".
- [ ] Pages deploy workflow updated to publish the new SPA.
