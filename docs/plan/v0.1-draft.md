# ACT — Agent Content Tree

**A specification for serving structured, progressively-disclosable content to LLM agents — across markdown sites, CMS-driven sites, component-driven applications, and authenticated SaaS apps. Static and runtime delivery profiles share one wire format.**

## 1. The problem we're solving

Existing options for serving content to LLM agents fall into a few buckets, none of which fit modern web pipelines that want progressive disclosure across heterogeneous content sources and across both public and authenticated contexts:

| Option | What it does | Why it falls short |
|---|---|---|
| `llms.txt` | Markdown index of links | A directory, not the content |
| `llms-full.txt` | Concatenated Markdown | Flat blob, no structure, all-or-nothing |
| Schema.org JSON-LD | Entity descriptions | For search, not document trees |
| MCP resources | Hierarchical content via protocol | Requires a running server |
| NLWeb | Queryable interface | Query-time, requires vector DB |

What's missing: **a static, build-time-generated, AST-shaped content tree with progressive disclosure semantics.** Think "RSS for LLM agents, but with full content and structure preserved."

ACT (Agent Content Tree) is a proposal for that format.

## 2. Design exploration: seven perspectives

### 2.1 Perspective A — The Format Architect

*"Treat the site as a tree of typed nodes, each with stable IDs, summary/abstract/full disclosure levels, and child references. Serve as static JSON files."*

Core proposal:

- **Manifest** at `/.well-known/act.json` — points to index, declares version, lists capabilities.
- **Index** at `/act/index.json` — flat list of every node with summaries, types, token estimates, ETags, and parent/child links. Cheap to fetch and scan.
- **Nodes** at `/act/n/{id}.json` — the full content of a single node, with child IDs to walk further.
- **Subtree bundles** at `/act/sub/{id}.json` — optional, returns a node and all descendants flattened, for bulk fetch.

Content blocks inside nodes use Markdown for prose, but typed blocks for code, tables, images, callouts, and references — so the agent doesn't have to re-parse Markdown to find structure.

Disclosure is per-node and explicit: every node carries a `summary` (≤1 sentence), an optional `abstract` (≤1 paragraph), and a `body` (full content). An agent can scan the index, read summaries, then fetch only the bodies it needs.

### 2.2 Perspective B — The LLM Agent (Consumer)

*"What I actually want when I land on a site I've never seen before."*

What's painful today:
1. I don't know what's on the site without scraping the whole thing.
2. When I scrape HTML, 80% of tokens are nav/footer/ads/scripts.
3. I can't tell which page is canonical for a given concept.
4. I can't cache anything safely because I don't know when content changed.
5. I can't budget — I don't know if "read the docs" is 5K tokens or 500K.

What I want:
1. **A map first.** One small fetch that tells me what exists, what type each thing is, how big it is, and how things relate.
2. **Stable identifiers.** Let me cache by ID + ETag.
3. **Token estimates.** Tell me what each node will cost me at each disclosure level so I can plan.
4. **Typed semantics.** "This is API reference, this is a tutorial, this is a marketing page" — let me filter.
5. **Cross-references as IDs, not URLs.** When the tutorial says "see the auth section," give me a node ID I can resolve, not a URL I have to re-fetch and parse.
6. **Code blocks I can trust.** Don't make me regex Markdown for fenced code.

Wishlist: an explicit "next sensible chunk" hint per node — if I'm reading X, what should I probably load alongside it?

### 2.3 Perspective C — The Author/Publisher

*"I don't want a second source of truth. I want this generated from my existing Markdown/MDX/CMS."*

Concerns:
- I already have docs in Markdown with frontmatter. Don't make me rewrite them.
- I want to author once and publish to humans (HTML) and agents (ACT) from the same source.
- I want sensible defaults — summaries auto-extracted from the first paragraph or `<meta description>`, IDs derived from URL slugs, types inferred from folder structure.
- I want to override anything via frontmatter when defaults are wrong.
- I do NOT want to maintain a JSON file by hand.

Implication: ACT must be generated by build tools (Next.js, Astro, Hugo, 11ty, Docusaurus, Mintlify, MkDocs, etc.) from existing content. The format must be friendly to that. Frontmatter conventions matter as much as the JSON schema.

### 2.4 Perspective D — The Build Engineer

*"I'm the one writing the generator. What's tractable?"*

Constraints:
- A static site can have 10K+ pages. The index has to be cheap to produce and serve.
- Token estimation needs a tokenizer — heavy at build time. Need approximation: char count / 3.5 is fine as a default. Allow exact tokenizer plug-in.
- ETag must be deterministic from content (hash), not from build time, or every rebuild invalidates everything.
- Should support partial regeneration — only rebuild changed nodes.
- Must work with CDNs: all files static, cacheable, immutable per ETag.
- Want a streaming/NDJSON variant of the index for very large sites.

Bonus: an agent-friendly sitemap format that pure search engines can also consume? Probably out of scope, but consider compatibility with sitemap.xml conventions.

### 2.5 Perspective E — The Skeptic

*"Why does this need to exist at all? You're reinventing wheels."*

Hard questions:

1. **"Just use MCP."** MCP requires a running server. ACT is for static sites and build pipelines. They compose: an MCP server can wrap an ACT tree as resources. ACT is the file format; MCP is the protocol.

2. **"Just use llms-full.txt."** That's a flat blob. No structure, no caching granularity, no token budgeting, no typed nodes, all-or-nothing fetches. Fine for tiny sites; fails at scale.

3. **"JSON-LD already exists."** Schema.org is for entities (person, product, article-as-thing) — it doesn't model document hierarchy or progressive disclosure. ACT can *reference* Schema.org types in its `metadata.schema_org` field but isn't a substitute.

4. **"Adoption will be zero."** Probably true unless (a) it ships as a default in popular SSGs, (b) it's parseable trivially with no required tooling, (c) major LLM clients/crawlers actually look for it. Strategy: keep the format almost insultingly simple; piggyback on llms.txt for discovery; provide a reference generator for Astro/Next/Hugo on day one.

5. **"JSON is token-expensive."** Real. Mitigations: short keys, optional fields default to omitted, summary-only fetches keep the wrapper small, and the wrapper amortizes well over the actual content payload. Optionally support YAML or a TOON-style serialization in v1+.

6. **"Won't this rot?"** Yes, like any spec. Mitigations: version field on every file, declare backward compat policy, keep the core minimal so extensions are obvious.

7. **"What about non-public content?"** ACT files can sit behind auth like any other resource. The manifest can declare authentication requirements. Sensitive content: don't put it in your public ACT, same as you wouldn't put it in your public sitemap.

8. **"Why not just better Markdown?"** Markdown alone has no schema, no IDs, no token estimates, no caching primitives, no progressive disclosure. ACT uses Markdown *inside* the structure for prose content — best of both.

### 2.6 Perspective F — The React Marketing Developer

*"You're describing a doc site. My world is different."*

Context: I work on a corporate marketing site. Next.js App Router, ~120 routes, four locales (en/es/de/ja), Contentful for most copy, Storyblok for the blog, react-intl for UI strings, ~40 design-system "block" components (Hero, FeatureGrid, PricingTable, TestimonialCarousel, FAQAccordion, CTA, etc.) composed into pages. We do A/B testing and have personalization variants. None of my content is in `.md` files.

The bones I have to pick with v0.1:

1. **"My content isn't files."** It's CMS entries, translation messages, and component props composed at runtime. Your "Markdown source → ACT" pipeline assumes a world I don't live in.

2. **"Your type taxonomy ignores marketing."** `article`, `tutorial`, `reference` — none of those describe a hero section, a feature grid, or a testimonial carousel. The unit of content on my pages isn't "a document," it's "a stack of typed sections."

3. **"You assume one body per page."** A landing page is hero + features + testimonials + pricing + FAQ + CTA. Six sections, six purposes, often six different content sources. Treating it as a single Markdown blob is a lie.

4. **"i18n isn't an afterthought."** I have four locales. The English `/pricing` relates to the Spanish `/precios`. Your spec mentions locales as an open question — for me they're a day-one requirement.

5. **"You're forcing a second source of truth."** I will never maintain content twice. If the spec doesn't generate ACT from my existing CMS + i18n + components, it's dead to me.

6. **"Stable IDs from URL slug breaks for dynamic routes."** `/products/[slug]` doesn't have a meaningful slug at build time without pulling from the CMS. ID derivation has to know about my data sources.

7. **"My design system is the schema."** I already have ~40 components that encode the structure of every marketing page on the site. That structure should *be* the ACT contract, declared once on each component, and emitted automatically wherever the component is used.

What I actually need:

- **Source adapters.** Plug-ins that read from CMS, i18n files, and component trees — not just files on disk.
- **A component instrumentation API.** Let my design-system components declare their ACT type and extraction logic once; have the build collect them automatically wherever they're rendered.
- **A first-class i18n model.** Per-locale trees, cross-locale `translation_of` references, locale negotiation in the manifest.
- **CMS schema mapping.** Declarative config: "Contentful content type `LandingPage` → ACT `page` node, field `hero` → `marketing:hero` content block, etc."
- **A render-time hook for SPAs.** For pages whose content composes only at runtime, let me run a build step that pre-renders into ACT — like prerendering for SEO, but emitting ACT instead of HTML.

If the spec accommodates all of that, ACT becomes a thing my pipeline emits as a side effect of work I'm already doing for SEO and i18n. If it doesn't, I'll never adopt it — and neither will any of the thousand other corporate sites built like mine.

### 2.7 Perspective G — The B2B SaaS Engineer

*"My content is per-tenant, behind auth, and changes constantly. There's no build step."*

Context: I work on a B2B SaaS — think Linear, Notion, Asana, ClickUp, an internal CRM. Tens of thousands of customer workspaces. Each customer logs in and sees their own data: projects, documents, tickets, messages. There is no "site I publish"; there are millions of per-tenant content trees that exist only when someone asks for them.

Where I sit relative to v0.1:

1. **"Build-time generation is meaningless to me."** I don't have a build that produces my customers' content. Their content is in Postgres and gets rendered on demand.

2. **"Auth is the whole game."** Every request is authenticated. The user gets a content tree scoped to their permissions. A static manifest at `/.well-known/act.json` either has to be auth-gated (then static caching is dubious) or has to be empty.

3. **"My customers want their AI assistants to read their workspace."** When a Notion user asks Claude "summarize the project I was working on yesterday," Claude needs scoped, authenticated, real-time access to that user's content. Today the answer is "build an MCP server."

4. **"MCP is great but heavy."** Implementing MCP means a separate transport (stdio or SSE), a separate auth flow (OAuth dance to connect the integration), a separate discovery layer. For "just let agents read my content via HTTP," it's overkill.

5. **"My users are already logged in."** When a browser-based agent (Claude in Chrome, a Copilot extension) is on my page, it can use the user's existing session cookies. Plain HTTP + JSON would Just Work. MCP wouldn't.

6. **"I'd love to use the same shape as static ACT."** My marketing site might already publish static ACT for `/pricing`, `/docs`, etc. If my authenticated app can serve the same JSON shape from a runtime endpoint, agents that know how to consume one know how to consume the other. One mental model, two delivery modes.

What I want:

- **A runtime profile of ACT.** Same wire format, served dynamically per request, with standard HTTP auth (cookies, Bearer tokens, OAuth — whatever I'm already using).
- **A small runtime SDK** for my framework (Next.js API routes, Express, FastAPI, Rails) so I can implement a few resolver functions and get ACT endpoints for free.
- **Clear guidance on ACT vs MCP.** When does my SaaS want one, the other, or both?

What this is *not*:

- A replacement for MCP. If my customer wants their agent to *create* a document or *complete* a task, MCP is right. ACT runtime is for read-only structured content access.
- A new auth scheme. I have OAuth and session cookies. Don't make me invent another one.

## 3. Critique round

Architect's design challenged by Skeptic's adoption concern → simplified manifest to require only 3 fields. Optional fields stay optional.

Agent's wish for "next sensible chunk" challenged by Build Engineer ("how do I compute that automatically?") → resolved as optional `related` field, populated by author or by simple heuristics (sibling, parent, frequently-co-cited).

Author's "don't make me hand-write JSON" challenged by Architect ("but the JSON IS the format") → resolved: ACT is the *output* format, not the authoring format. Authors write Markdown + frontmatter; build tools emit ACT.

Build Engineer's concern about ETags + 10K pages challenged Architect's "monolithic index" → resolved by adding optional NDJSON-streamed index variant (`index.ndjson`) and per-section sub-indexes.

Skeptic's "JSON is token-expensive" challenged disclosure model → resolved: the index ships only summaries by default; full bodies are fetched per-node. The wrapper is paid for once per node, not per token of content.

LLM Agent's wish for "stable cross-references" challenged by Author's "I don't want to assign IDs by hand" → resolved: IDs are derived from URL slug by default, overridable via frontmatter. Stability is the build tool's responsibility — same input → same ID.

Architect's "typed content blocks" challenged by Skeptic ("over-engineering") → kept, but the type vocabulary is small (prose, code, table, image, callout, reference) and extension is namespaced. Markdown bodies are still allowed as a fallback content type.

### 3.1 Critique round 2 — component-driven content

The React Marketing Developer's perspective forces several v0.1 assumptions to break:

**"Markdown is the source format."** Wrong assumption. Markdown is *one* source format; the spec must be source-agnostic. Resolution: introduce a **source adapter** abstraction. The build is `[adapters] → normalized node stream → emitter`. v0.1 ships reference adapters for markdown, mdx, contentful, sanity, storyblok, strapi, builder.io, react-intl/next-intl/i18next, and React component trees. Custom adapters are namespaced.

**"One body per page."** Too restrictive for component-composed pages. Resolution: keep the single `content` array per node, but lean heavily on typed content blocks. A landing page is a single node whose `content` is `[hero-block, feature-grid-block, testimonials-block, pricing-block, faq-block, cta-block]`. Each block carries its own block-level type, summary, and structured payload. Reusable sections (e.g., a global FAQ used across pages) can be standalone nodes referenced from multiple pages via `reference` blocks.

**"Document type taxonomy ignores marketing."** Correct. Resolution: extend the v0.1 node taxonomy with `page`, `section`, `collection`, and a reserved `marketing:` namespace for block types. Block types expand to cover marketing primitives (`marketing:hero`, `marketing:feature-grid`, `marketing:pricing-table`, etc.).

**"i18n is an open question."** Was. Resolution: promote i18n to first-class. Manifest declares available locales; per-locale trees live at predictable paths; nodes carry `translation_of` references. Day-one requirement, not a deferral.

**"ID derivation breaks for dynamic routes."** Correct for any data-driven route. Resolution: ID strategies expand from `slug | path | hash | custom` to also include `cms_id` (use the CMS entry ID), `route_with_params` (resolve dynamic params at build time using the data layer), and `composite` (stable hash of source adapter + key path).

**"How do components declare their ACT contract?"** New question. Resolution: define a small **component instrumentation API** with three equivalent declaration patterns (static field, decorator, hook). Build tooling extracts the contract via static analysis (babel/SWC plugin) or render-time collection (server-render walk) — typically both, used together.

**"How do we handle SPAs whose content composes only at runtime?"** New question. Resolution: render-time collection. The build runs `react-dom/server` (or framework equivalent) against each route, with an `<ActProvider>` at the root that captures component contracts as the tree renders. Dynamic content gets resolved against real data; the output is the same static ACT file an SSG would have produced.

### 3.2 Critique round 3 — runtime delivery and the MCP question

The B2B SaaS Engineer's perspective forces a final reframe:

**"Build-time generation is meaningless for per-tenant content."** Correct. Resolution: separate the **wire format** (what the JSON looks like) from the **delivery profile** (where it comes from). v0.1 defines two profiles sharing one format:
- **Static profile** — files written at build time, served via CDN. Everything described so far.
- **Runtime profile** — endpoints that respond with the same JSON shapes per request, with HTTP auth and cache directives.

The format is identical. Agents that consume one consume the other.

**"Isn't this just MCP with extra steps?"** No, but the overlap is real and worth being precise about.

| Concern | ACT (runtime) | MCP |
|---|---|---|
| Read structured content | ✅ optimized for this | ✅ via resources |
| Mutate state / take actions | ❌ read-only | ✅ via tools |
| Transport | Plain HTTP + JSON | JSON-RPC (stdio / SSE / HTTP) |
| Auth | Whatever HTTP supports (cookies, Bearer, OAuth) | OAuth flow, per-server connection |
| Browser-agent fit | Excellent (uses existing session) | Workable but heavier |
| Subscriptions / push | ❌ request/response only | ✅ notifications |
| Wire format = static profile | ✅ identical | ❌ different shape |
| Implementation cost | Trivial — it's just an API endpoint | Moderate — implement the protocol |

The honest framing: **ACT runtime is the read-only HTTP-native sibling of MCP resources.** They compose:
- A SaaS app might serve **ACT runtime** for "let agents read my content via HTTP using the user's session"
- And expose **MCP tools** for "let agents take actions or subscribe to changes"
- And publish **static ACT** for the public marketing site, blog, and docs

All three speak shapes that share lineage. Agents that know ACT consume static and runtime equally. Agents that need actions or push updates use MCP alongside.

**"Won't auth break caching?"** Per-user content is less cacheable, true. Resolution: ACT runtime uses standard HTTP cache directives (`Cache-Control: private` for user-scoped content, `public` for tenant-shared, `Vary: Authorization` where needed). ETags still work — they just key off (content + auth identity) instead of content alone. Agents revalidate with `If-None-Match` like any HTTP client.

**"Do we need a runtime SDK?"** Reference SDKs for major frameworks (Next.js route handlers, Express, FastAPI, Rails) are essential for adoption. The SDK pattern is small: define resolver functions for `manifest`, `index`, `node`, `subtree`; the SDK handles routing, ETag computation, caching headers, and error responses.

**"Does runtime mode change the format?"** No. The JSON shapes in §5.3–5.7 are identical for both profiles. Runtime adds three things on top: HTTP semantics for fetch/cache/auth (§5.13), an SDK pattern for resolver functions (§6.6), and a decision matrix for when to use what (§5.14).

## 4. Resolved design decisions

| Decision | Choice | Rationale |
|---|---|---|
| Format | JSON (with optional YAML serialization) | Universal parseability; matches existing tooling |
| Content payload | Markdown by default, typed blocks optional | LLMs already excellent at Markdown; typed blocks for structure |
| Discovery | `/.well-known/act.json` + reference from `/llms.txt` | Standard well-known location; bridges existing convention |
| Hierarchy | Tree with cross-references (DAG in practice) | Documents are mostly trees; cross-refs handle the rest |
| IDs | Derived from URL path; overridable | Stable across builds; human-debuggable |
| Disclosure levels | summary / abstract / body | Three levels covers 95% of needs without complexity |
| Caching | ETag per node, deterministic from content hash | CDN-friendly, rebuild-stable |
| Token estimates | Required field, char-count approximation OK | Lets agents budget without running a tokenizer |
| Bulk fetch | Optional subtree endpoints | Static-friendly; agents can opt in |
| Auth | Out of scope; treat as standard HTTP | No new mechanism needed |
| Versioning | `act_version` field at every level | Cheap insurance |
| Source model | Pluggable adapters; not file-based | Marketing/SPAs/CMS-driven sites have no markdown source |
| Page composition | Multi-block content arrays per node | Landing pages aren't single documents |
| i18n | First-class; per-locale trees with cross-locale refs | Multi-locale sites are common, not edge cases |
| Component contract | Three equivalent declaration patterns (field/decorator/hook) | Different teams have different style preferences |
| ID strategies | Add `cms_id`, `route_with_params`, `composite` | Dynamic routes need data-layer-aware IDs |
| Marketing types | New `marketing:` block namespace + `page`/`section`/`collection` node types | Document-only taxonomy was too narrow |
| SPA pipeline | Render-time collection via `<ActProvider>` during server render | Required for any app whose content composes at runtime |
| Delivery profiles | Two: **static** (build-time files) and **runtime** (HTTP endpoints), sharing one wire format | Per-tenant SaaS can't generate at build time |
| Runtime auth | Standard HTTP (cookies, Bearer, OAuth) — no new scheme | Reuse what apps already have |
| Runtime caching | HTTP ETags + Cache-Control + Vary | Standard, CDN-friendly |
| ACT vs MCP | Complementary; ACT for read-only content, MCP for tools/actions/push | Different jobs, compose well |

## 5. Specification

### 5.1 Overview

An ACT-published site exposes:

1. A **manifest** at `/.well-known/act.json`.
2. An **index** at the URL given by the manifest (default `/act/index.json`).
3. Individual **node files** at the URL template given by the manifest (default `/act/n/{id}.json`).
4. Optional **subtree files** at `/act/sub/{id}.json`.

All files are static JSON. All fields are UTF-8.

### 5.2 Discovery

An agent's discovery flow:

```
1. Fetch  /.well-known/act.json
2. Read   manifest, get index_url and node_url_template
3. Fetch  index_url
4. Scan   summaries, types, token estimates
5. Fetch  individual node files as needed (substitute id into node_url_template)
6. Cache  by ETag
```

Sites SHOULD also reference their ACT manifest from `/llms.txt`:

```markdown
# Acme Docs

> Documentation for Acme widgets.

This site provides an ACT (Agent Content Tree) feed:
- [ACT Manifest](/.well-known/act.json)
```

### 5.3 Manifest

**Location:** `/.well-known/act.json`

**Required fields:**

```json
{
  "act_version": "0.1",
  "site": {
    "name": "Acme Docs"
  },
  "index_url": "/act/index.json",
  "node_url_template": "/act/n/{id}.json"
}
```

**Full example:**

```json
{
  "act_version": "0.1",
  "site": {
    "name": "Acme Docs",
    "description": "Documentation for Acme widgets",
    "canonical_url": "https://acme.com",
    "locale": "en-US",
    "license": "CC-BY-4.0"
  },
  "generated_at": "2026-04-30T12:00:00Z",
  "generator": "act-astro/0.3.1",
  "index_url": "/act/index.json",
  "index_ndjson_url": "/act/index.ndjson",
  "node_url_template": "/act/n/{id}.json",
  "subtree_url_template": "/act/sub/{id}.json",
  "search_url_template": "/act/search?q={query}",
  "root_id": "root",
  "stats": {
    "node_count": 247,
    "total_tokens_full": 184320,
    "total_tokens_summary": 5420
  },
  "capabilities": ["subtree", "ndjson_index", "etag"],
  "policy": {
    "robots_respected": true,
    "rate_limit_per_minute": 600,
    "contact": "agents@acme.com"
  }
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `act_version` | string | yes | Spec version (semver-ish; this draft is "0.1") |
| `site.name` | string | yes | Human-readable site name |
| `site.description` | string | no | Short description |
| `site.canonical_url` | string | no | Canonical site URL |
| `site.locale` | string | no | BCP-47 locale tag |
| `site.license` | string | no | SPDX identifier or URL |
| `generated_at` | ISO 8601 | no | Build timestamp |
| `generator` | string | no | Tool that produced this |
| `index_url` | string | yes | URL of the index file |
| `index_ndjson_url` | string | no | URL of NDJSON-streamed index for large sites |
| `node_url_template` | string | yes | URL template with `{id}` placeholder |
| `subtree_url_template` | string | no | URL template for subtree fetches |
| `search_url_template` | string | no | URL template for free-text search |
| `root_id` | string | no | ID of the root node (default: "root") |
| `stats` | object | no | Counts and token totals |
| `capabilities` | array | no | Optional features supported |
| `policy` | object | no | Rate limits and contact |

### 5.4 Index

**Location:** as declared in `manifest.index_url`.

The index lists every node with its summary-level metadata. It MUST NOT contain full bodies.

```json
{
  "act_version": "0.1",
  "generated_at": "2026-04-30T12:00:00Z",
  "etag": "sha256:9f2c1b...",
  "nodes": [
    {
      "id": "intro",
      "type": "article",
      "title": "Introduction",
      "path": ["intro"],
      "summary": "An overview of Acme widgets and what you can build with them.",
      "tokens": { "summary": 14, "abstract": 80, "body": 480 },
      "etag": "sha256:abc123...",
      "updated_at": "2026-04-15T08:00:00Z",
      "parent": null,
      "children": ["intro/getting-started", "intro/concepts"],
      "tags": ["overview", "beginner"]
    },
    {
      "id": "intro/getting-started",
      "type": "tutorial",
      "title": "Getting started",
      "path": ["intro", "getting-started"],
      "summary": "Install the SDK and send your first request in 5 minutes.",
      "tokens": { "summary": 13, "body": 920 },
      "etag": "sha256:def456...",
      "parent": "intro",
      "children": []
    }
  ]
}
```

**NDJSON variant** (`index_ndjson_url`): one node entry per line, same fields. Suitable for sites with thousands of nodes; agents can stream-parse.

### 5.5 Node format

**Location:** as resolved from `manifest.node_url_template`.

```json
{
  "act_version": "0.1",
  "id": "intro/getting-started",
  "type": "tutorial",
  "title": "Getting started",
  "etag": "sha256:def456...",
  "updated_at": "2026-04-15T08:00:00Z",

  "summary": "Install the SDK and send your first request in 5 minutes.",
  "abstract": "This tutorial walks you through installing the Acme SDK in Node.js or Python, configuring authentication, and making your first widget request.",

  "content": [
    {
      "type": "prose",
      "format": "markdown",
      "text": "## Install\n\nFirst, install the SDK..."
    },
    {
      "type": "code",
      "language": "bash",
      "text": "npm install @acme/sdk"
    },
    {
      "type": "callout",
      "level": "warning",
      "text": "Node 18 or higher is required."
    }
  ],

  "tokens": { "summary": 13, "abstract": 42, "body": 920 },

  "parent": "intro",
  "children": [],
  "related": ["concepts/authentication", "reference/widgets-api"],

  "source": {
    "human_url": "https://acme.com/docs/getting-started",
    "edit_url": "https://github.com/acme/docs/edit/main/getting-started.md"
  },

  "metadata": {
    "tags": ["beginner", "tutorial"],
    "audience": ["developers"],
    "schema_org_type": "TechArticle",
    "language": "en-US"
  }
}
```

#### 5.5.1 Required vs optional fields

**Required:** `act_version`, `id`, `type`, `title`, `etag`, `summary`, `content`, `tokens`.

**Optional:** everything else.

#### 5.5.2 Content block types

The `content` array is an ordered list of typed blocks. Defined block types:

| Type | Required fields | Notes |
|---|---|---|
| `prose` | `format`, `text` | `format` is "markdown" or "plain"; "markdown" is default |
| `code` | `language`, `text` | Language per linguist conventions |
| `table` | `rows` (2D array) | Optional `header` boolean for first row |
| `image` | `url` | Optional `alt`, `caption`, `width`, `height` |
| `video` | `url` | Optional `transcript`, `duration` |
| `callout` | `level`, `text` | `level`: "info", "warning", "tip", "danger" |
| `reference` | `to` (node id) | Cross-reference; optional `text` for inline label |
| `quote` | `text` | Optional `cite`, `cite_url` |
| `qa` | `question`, `answer` | Single Q&A pair (FAQ items) |
| `data` | `format`, `value` | Structured data; `format`: "json", "yaml", "csv" |
| `custom` | `namespace`, `kind`, `payload` | Extension mechanism |

**Marketing block types** (reserved `marketing:` namespace, intended for component-driven sites):

| Type | Common fields | Notes |
|---|---|---|
| `marketing:hero` | `headline`, `subhead`, `cta` | Top-of-page promotional block |
| `marketing:feature-grid` | `features[]` (array of `{title, description, icon?}`) | Feature highlights |
| `marketing:testimonial` | `quote`, `author`, `role`, `org` | Single testimonial; multiples via `data` block |
| `marketing:pricing-table` | `tiers[]` (array with `name`, `price`, `features`) | Pricing comparison |
| `marketing:faq` | `items[]` (array of `qa` blocks) | Page-scoped FAQ |
| `marketing:cta` | `headline`, `body`, `actions[]` | Call-to-action band |
| `marketing:logo-cloud` | `logos[]` (with `alt`, `url?`) | Trusted-by/customer logos |
| `marketing:stats` | `stats[]` (with `value`, `label`) | Numeric stat callouts |
| `marketing:contact` | `methods[]` | Contact info / form |
| `marketing:nav` | `links[]` | Site navigation, when relevant for context |

Marketing block types are advisory — agents that don't recognize them treat them as opaque structured payloads and can still extract any embedded `prose`, `text`, or `headline` fields.

A node MAY use a single `prose` block for its entire body if no structure is needed. The format degrades gracefully to "Markdown in JSON."

#### 5.5.3 The `type` field (node taxonomy)

Reserved node types in v0.1, organized by category:

**Document types:**
- `article` — general informational content
- `tutorial` — step-by-step guidance
- `reference` — API/spec documentation
- `concept` — explanatory/conceptual content
- `faq` — question-and-answer
- `policy` — legal, terms, privacy
- `glossary` — defined-term entry
- `post` — blog post / news

**Page types** (for component-composed sites):
- `page` — generic web page; content is an ordered list of typed blocks
- `section` — a reusable page section (referenced from multiple pages)
- `landing` — marketing/promotional page (specialization of `page`)
- `collection` — listing/index page

**Entity types:**
- `product` — product/service description
- `index` — collection/listing of other nodes (e.g., a docs root)

Custom types MUST be namespaced: `"type": "acme:changelog-entry"`.

### 5.6 Progressive disclosure

The disclosure model is three levels per node:

1. **Summary** (≤1 sentence) — present in the index. ~10–20 tokens.
2. **Abstract** (≤1 paragraph) — present in the node file. ~30–80 tokens. Optional.
3. **Body** — full `content` array. Variable.

Agents typically:

- Fetch the **index** to scan summaries.
- Fetch a **node file** when a summary looks relevant.
- Walk **children** by ID, repeating step 2.
- Use **subtree fetch** when a whole section is clearly needed.

### 5.7 Bulk operations

If the manifest declares the `subtree` capability, an agent MAY fetch a flattened subtree:

`GET /act/sub/{id}.json`

```json
{
  "act_version": "0.1",
  "root": "intro",
  "etag": "sha256:...",
  "tokens": { "body": 4200 },
  "nodes": [
    { /* full node object for "intro" */ },
    { /* full node object for "intro/getting-started" */ },
    { /* etc */ }
  ]
}
```

### 5.8 Caching & versioning

- Every file carries an `etag` (recommended: `sha256:` prefix + content hash).
- Servers SHOULD send HTTP `ETag` headers matching the field value.
- Agents SHOULD use `If-None-Match` for revalidation.
- ETags MUST be deterministic from content — same content yields the same ETag across rebuilds.
- Node ETags MUST change when the node's `content`, `summary`, or `abstract` changes.
- Index ETag MUST change when any node's metadata changes.

### 5.9 Search (optional)

If `manifest.search_url_template` is set, agents can perform free-text queries:

`GET /act/search?q={url-encoded-query}`

Returns a search result envelope:

```json
{
  "act_version": "0.1",
  "query": "authentication",
  "hits": [
    { "id": "concepts/authentication", "score": 0.92, "snippet": "..." },
    { "id": "reference/auth-api", "score": 0.78, "snippet": "..." }
  ]
}
```

This is purely a convenience — implementations can stub it with grep, vector search, or anything in between.

### 5.10 Source adapters

ACT is source-agnostic. The build pipeline reads from one or more **source adapters**, each contributing nodes to the final tree.

#### 5.10.1 Adapter contract

```typescript
interface ActSourceAdapter {
  name: string;                              // "markdown", "contentful", etc.
  init(config: AdapterConfig, ctx: BuildContext): Promise<void>;
  collect(): AsyncIterable<NodeDraft>;
  watch?(onChange: (id: string) => void): void;  // optional: dev-server support
}

interface NodeDraft {
  id?: string;             // optional; generator assigns from id_strategy if absent
  source: string;          // adapter origin, e.g. "contentful:abc123"
  type?: string;
  title: string;
  summary?: string;
  abstract?: string;
  content: ContentBlock[];
  parent?: string;
  children?: string[];
  related?: string[];
  metadata?: Record<string, any>;
  locale?: string;         // BCP-47
  translation_of?: { locale: string; id: string };
}
```

A draft is incomplete; the generator fills in `etag`, `tokens`, normalizes `id`, validates against the schema, and emits the final node file.

#### 5.10.2 Reference adapters in v0.1

| Adapter | Reads from | Notes |
|---|---|---|
| `markdown` | `*.md` files | Frontmatter under `act:` key |
| `mdx` | `*.mdx` files | Same as markdown; JSX is stripped or rendered to prose |
| `contentful` | Contentful Delivery API | Schema mapping config required |
| `sanity` | Sanity GROQ queries | Schema mapping config required |
| `storyblok` | Storyblok CDN | Component-aware (Storyblok blocks → ACT blocks) |
| `strapi` | Strapi REST/GraphQL | Schema mapping config required |
| `builder` | Builder.io spaces | Visual builder integration |
| `i18n` | next-intl, react-intl, i18next message catalogs | See §5.12 |
| `react-component` | React component tree | See §5.11 |
| `vue-component` | Vue component tree | Analogous to React |
| `angular-component` | Angular component tree | Analogous to React |
| `programmatic` | Arbitrary JS/TS function returning `NodeDraft[]` | Escape hatch |

Custom adapters MUST be namespaced (`acme:internal-cms`).

#### 5.10.3 Multi-source merging

When multiple adapters contribute drafts for the same `id`, the generator follows a configurable resolution policy:

- `error` (default) — fail the build; misconfiguration.
- `first` — first adapter to produce the ID wins.
- `last` — last adapter wins (useful for overrides).
- `merge` — shallow-merge fields; arrays concatenate; later sources win on scalar conflicts.

This handles the common case where, e.g., a CMS adapter provides the page node and an i18n adapter contributes UI strings as additional content blocks.

### 5.11 Component instrumentation

For React/Vue/Angular apps where content lives in components, ACT defines a small instrumentation API. Components declare their ACT contract once; build tooling extracts it at build time.

#### 5.11.1 Three equivalent declaration patterns

**Pattern 1 — Static field** (recommended for design systems):

```jsx
// design-system/Hero.jsx
export function Hero({ title, subtitle, ctaText, ctaUrl }) {
  return <section className="hero">{/* ... */}</section>;
}

Hero.act = {
  type: 'marketing:hero',
  extract: (props) => ({
    summary: props.title,
    content: [{
      type: 'marketing:hero',
      headline: props.title,
      subhead: props.subtitle,
      cta: props.ctaText ? { text: props.ctaText, to: props.ctaUrl } : undefined
    }]
  })
};
```

**Pattern 2 — Hook** (for components with derived/contextual state):

```jsx
import { useActBlock } from '@act/react';

export function PricingTable({ tiers, highlightedTier }) {
  useActBlock({
    type: 'marketing:pricing-table',
    extract: () => ({
      summary: `Pricing for ${tiers.length} tiers: ${tiers.map(t => t.name).join(', ')}.`,
      content: [{
        type: 'marketing:pricing-table',
        tiers,
        highlighted: highlightedTier
      }]
    })
  });
  return <div className="pricing">{/* ... */}</div>;
}
```

**Pattern 3 — Decorator** (TypeScript / class components):

```typescript
@actBlock({
  type: 'marketing:faq',
  extract: (props: FAQProps) => ({
    summary: `${props.items.length} frequently asked questions.`,
    content: [{
      type: 'marketing:faq',
      items: props.items.map(i => ({ type: 'qa', question: i.q, answer: i.a }))
    }]
  })
})
export class FAQAccordion extends React.Component<FAQProps> { /* ... */ }
```

All three patterns produce equivalent results; teams pick whichever fits their style.

#### 5.11.2 Page-level contracts

Routes/pages can declare their own `act` contract to set node-level fields (type, summary, related). These compose with block contracts from child components:

```jsx
// app/pricing/page.jsx
export default function PricingPage() {
  return (
    <>
      <Hero title="Pricing" subtitle="..." />
      <PricingTable tiers={...} />
      <FAQAccordion items={...} />
      <CTA headline="..." />
    </>
  );
}

PricingPage.act = {
  type: 'landing',
  id: 'pricing',
  summary: 'Acme pricing tiers, plan comparison, and FAQ.',
  related: ['products', 'contact']
};
```

The resulting node has its `content` populated automatically from the contracts of `Hero`, `PricingTable`, `FAQAccordion`, and `CTA`, in render order.

#### 5.11.3 Build-time extraction strategies

Two strategies, used independently or together:

**Static analysis** (babel/SWC plugin). Walks imports and JSX, finds components with `.act` field or `@actBlock` decorator, evaluates the `extract` function against props known at build time. Fast; no runtime; limited to statically resolvable extractions.

**Render-time collection** (Node SSR pass). The build runs `react-dom/server.renderToString` against each route, with an `<ActProvider>` at the root. The provider collects:
- `Component.act.extract(props)` outputs as components mount
- `useActBlock` registrations from hooks
- Page-level `Page.act` contracts

After the render, the provider yields a `NodeDraft` per route. Handles arbitrarily dynamic content (data fetching, branching, personalization defaults). Requires the app to be SSR-able — a constraint most modern React apps already satisfy.

**Recommended:** static analysis for the design system (where contracts are clean), render-time for pages (where routing and data resolution happen).

#### 5.11.4 Variant handling (A/B tests, personalization)

Components rendered conditionally for personalization or A/B variants pose a question: which variant ends up in ACT?

The default rule: **the canonical variant is whatever the build renders for the default-audience, default-experiment-arm, default-locale request.** Other variants are discarded. Authors who need to expose variants explicitly can use a multi-variant block:

```jsx
Hero.act = {
  type: 'marketing:hero',
  variants: 'all',  // emit all variants instead of just the default
  extract: (props, ctx) => ({ /* ... */ })
};
```

When `variants: 'all'`, the build runs the page render once per variant and emits a parallel set of nodes with `metadata.variant_of` references.

#### 5.11.5 Vue and Angular

Vue components use `<script setup>` `defineActBlock(...)` or a static `act` export. Angular components use `@ActBlock(...)` decorators. Semantics are identical to React's; only the binding mechanism differs.

### 5.12 Internationalization

ACT treats locale as first-class.

#### 5.12.1 Locale model in the manifest

```json
{
  "act_version": "0.1",
  "site": { "name": "Acme", "locale": "en-US" },
  "locales": {
    "default": "en-US",
    "available": ["en-US", "es-ES", "de-DE", "ja-JP"],
    "manifest_url_template": "/{locale}/.well-known/act.json"
  },
  "index_url": "/act/index.json",
  "node_url_template": "/act/n/{id}.json"
}
```

Each non-default locale has its own manifest, index, and node files at the templated path. The default-locale manifest at `/.well-known/act.json` is the canonical entry point and points to siblings via `locales.manifest_url_template`.

#### 5.12.2 Cross-locale references

Each translated node carries a `translation_of` field referencing the default-locale node:

```json
{
  "id": "precios",
  "type": "landing",
  "title": "Precios",
  "locale": "es-ES",
  "translation_of": { "locale": "en-US", "id": "pricing" },
  "summary": "Niveles de precios de Acme y comparación de planes."
}
```

This lets agents fetch the right locale and confirm translation lineage. The default-locale node carries no `translation_of`.

#### 5.12.3 The `i18n` source adapter

Most React/Vue/Angular sites have UI strings in message catalogs (`en.json`, `es.json`, etc.) consumed by libraries like `next-intl`, `react-intl`, `i18next`, `vue-i18n`, or Angular's i18n. The `i18n` source adapter reads these directly:

```jsonc
// messages/en.json (next-intl style)
{
  "pricing": {
    "headline": "Simple, transparent pricing",
    "subhead": "No hidden fees. Cancel anytime.",
    "tiers": {
      "starter": { "name": "Starter", "price": "$9/mo" },
      "pro": { "name": "Pro", "price": "$29/mo" }
    }
  }
}
```

The adapter contributes nodes per locale, namespace by namespace, with content reflecting the catalog structure. When combined with a `react-component` adapter, the build cross-links message keys to the routes that consume them, so each route's ACT node inherits the relevant translated copy.

#### 5.12.4 Untranslated keys

When a locale lacks a translation for a given key, ACT generators fall back to the default-locale value and mark the affected block:

```json
{
  "type": "prose",
  "format": "markdown",
  "text": "...",
  "metadata": { "translation_status": "fallback", "fallback_from": "en-US" }
}
```

Agents that care about translation completeness can detect this and adjust behavior (e.g., warn the user, prefer the default-locale node).

### 5.13 Runtime serving (the dynamic profile)

ACT defines two delivery profiles that share one wire format:

- **Static profile** — files written at build time, served via CDN. Everything in §5.10–§5.12 describes this profile.
- **Runtime profile** — HTTP endpoints that respond with the same JSON shapes per request. Used for authenticated, per-tenant, or otherwise dynamic content. This is where SaaS apps live.

The wire format defined in §5.3–§5.9 is identical for both. An agent can consume either without changing its parsing code.

#### 5.13.1 The runtime contract

A runtime ACT server exposes four endpoints (paths configurable via the manifest):

| Endpoint | Method | Returns | Notes |
|---|---|---|---|
| `/.well-known/act.json` | GET | Manifest | May vary per authenticated user/tenant |
| `/act/index.json` | GET | Index | Scoped to the requester's accessible content |
| `/act/n/{id}.json` | GET | Node | Auth-checked per node |
| `/act/sub/{id}.json` | GET | Subtree | Optional; auth-checked |

Endpoints respond with the JSON envelopes already defined in §5.3 (manifest), §5.4 (index), §5.5 (node), §5.7 (subtree). No new shapes.

#### 5.13.2 Authentication

ACT runtime uses **whatever HTTP authentication the host application already uses**. The spec does not define a new auth scheme. Common patterns:

- **Session cookies** — perfect for browser-based agents that already have the user's session. The agent's fetch carries the cookie automatically.
- **Bearer tokens** — for API consumers, programmatic agents, headless integrations.
- **OAuth 2.0** — for third-party agents acting on behalf of the user.
- **API keys** — for service-to-service or trusted-tenant access.

The manifest MAY declare which auth schemes are supported:

```json
{
  "act_version": "0.1",
  "site": { "name": "Acme Workspace" },
  "delivery": "runtime",
  "auth": {
    "schemes": ["cookie", "bearer", "oauth2"],
    "oauth2": {
      "authorization_endpoint": "https://acme.com/oauth/authorize",
      "token_endpoint": "https://acme.com/oauth/token",
      "scopes_supported": ["act.read"]
    }
  },
  "index_url": "/act/index.json",
  "node_url_template": "/act/n/{id}.json"
}
```

Unauthenticated requests SHOULD return `401 Unauthorized` with a `WWW-Authenticate` header. Authenticated requests for content the user can't see SHOULD return `404 Not Found` (not `403`) to avoid leaking the existence of resources.

#### 5.13.3 Caching

Runtime ACT uses standard HTTP caching:

- **ETags** — every node and the index carry an `etag` field; servers MUST send it as the `ETag` HTTP header. Agents revalidate with `If-None-Match`.
- **Cache-Control** — `private` for user-scoped content, `public` for tenant-shared (or fully public) content, with `max-age` per the resource's volatility.
- **Vary** — when content varies by auth identity, send `Vary: Authorization` (or `Vary: Cookie` for cookie auth) so intermediaries don't serve the wrong user's content.

ETag derivation in runtime mode keys off `(content, auth_identity, tenant_id)` — same content for the same user yields the same ETag, but two users see different ETags for "the same" resource because their accessible content differs.

#### 5.13.4 Per-tenant scoping

A tenant-aware SaaS typically scopes every request by the authenticated user's tenant. The manifest, index, and nodes returned reflect *that user's* accessible content, not the global content set. The `id_strategy` for runtime nodes commonly uses `cms_id` or a composite of `(tenant_id, resource_id)` to keep IDs stable across requests for the same user without colliding across tenants.

#### 5.13.5 Hybrid sites: static + runtime in the same product

Most real SaaS products combine both:

- Public marketing site, blog, and public docs → **static** ACT, served by CDN.
- The authenticated app (workspaces, dashboards, user-scoped content) → **runtime** ACT, served by the application.

These can live at different paths under the same domain:

```
https://acme.com/.well-known/act.json          → public static manifest
https://app.acme.com/.well-known/act.json      → authenticated runtime manifest
```

Or via a single manifest that declares both via `mounts`:

```json
{
  "act_version": "0.1",
  "site": { "name": "Acme" },
  "mounts": [
    { "prefix": "/marketing", "delivery": "static", "manifest_url": "/marketing/.well-known/act.json" },
    { "prefix": "/app",       "delivery": "runtime", "manifest_url": "/app/.well-known/act.json" }
  ]
}
```

Agents follow `mounts` to the appropriate sub-manifest.

#### 5.13.6 Streaming and subscriptions (out of scope for v0.1)

Real-time updates (e.g., "tell me when this document changes") are not part of v0.1 runtime. Apps that need push semantics should expose those via MCP alongside ACT, or use webhooks/SSE in their own conventions. v0.2 may add an optional SSE-based change feed.

### 5.14 When to use static, runtime, MCP — or all three

ACT and MCP are complementary, not competitive. Different jobs, often shipped together. Decision matrix:

| You want to... | Use |
|---|---|
| Serve public docs, marketing pages, blog posts, product catalogs at scale | **Static ACT** |
| Let agents read per-tenant content via HTTP using the user's existing session | **Runtime ACT** |
| Serve the same content to humans (HTML) and agents (ACT) from one source | **Static ACT** with build adapters |
| Let agents take actions (create, update, delete, send) | **MCP tools** |
| Push real-time updates to agents | **MCP** (notifications) |
| Expose authenticated read-only content to a third-party AI client (ChatGPT, Claude, etc.) that connects via OAuth | Either **Runtime ACT** or **MCP resources** — pick based on the rest of your stack |
| Support browser-based agents that already have the user's session cookies | **Runtime ACT** (lowest friction; MCP requires connection setup) |
| Serve per-user *and* take actions on the user's behalf | **Runtime ACT** + **MCP** (use both) |

A typical mature deployment for a B2B SaaS:

```
acme.com (marketing)              → static ACT
docs.acme.com                     → static ACT
app.acme.com (authenticated app)  → runtime ACT (read-only content access)
                                  + MCP server (actions, subscriptions)
```

The marketing site, docs, and app all expose ACT in the same shape. Agents that understand ACT can read all three. The MCP server adds tool calling for users who connect their workspace to an agent platform.

### 6.2 Per-document frontmatter

Authors override defaults via standard YAML frontmatter:

```markdown
---
act:
  id: tutorials/auth/oauth-flow
  type: tutorial
  summary: Walk through the full OAuth 2.0 PKCE flow with Acme.
  abstract: |
    A step-by-step guide showing every redirect, parameter, and token
    exchange involved in authenticating an end user via OAuth 2.0 with
    PKCE. Includes copy-paste code for Node, Python, and Go.
  tags: [auth, oauth, pkce]
  related:
    - reference/auth-api
    - concepts/authentication
  exclude: false
---

# OAuth 2.0 with PKCE

The full flow involves...
```

The top-level `act:` namespace keeps ACT-specific frontmatter from colliding with other tools' conventions (Hugo, Jekyll, Astro all use frontmatter for their own purposes).

If a field is absent, the generator falls back to its strategy:
- `id` — derived from path/slug.
- `summary` — first sentence of body, or the page's `<meta description>` equivalent.
- `abstract` — first paragraph (skipping summary).
- `type` — inferred from `type_inference.by_path`.
- `tags` — from existing tag/category frontmatter if present.

### 6.3 Strategies

- **id_strategy**: `slug` (default), `path`, `hash`, `cms_id`, `route_with_params`, `composite`, `custom`
- **summary_strategy**: `frontmatter_or_first_paragraph` (default), `frontmatter_only`, `component_extract`, `llm_generated` (if API key configured)
- **tokenizer**: `approx` (chars/3.5, default), `tiktoken-cl100k`, `tiktoken-o200k`, `claude`, `custom`

### 6.6 Runtime SDK pattern (for SaaS apps)

Static-mode ACT is configured by `act.config.json`. Runtime-mode ACT is configured by writing **resolver functions** that the runtime SDK exposes as HTTP endpoints. The pattern is small and framework-agnostic; reference SDKs target Next.js, Express, FastAPI, Rails, Hono, and a generic WHATWG-fetch adapter.

#### 6.6.1 Next.js example

```typescript
// app/.well-known/act.json/route.ts
import { actManifestHandler } from '@act/runtime/next';

export const GET = actManifestHandler({
  resolveManifest: async (req) => ({
    act_version: '0.1',
    site: { name: 'Acme Workspace' },
    delivery: 'runtime',
    auth: { schemes: ['cookie', 'bearer'] },
    index_url: '/act/index.json',
    node_url_template: '/act/n/{id}.json'
  })
});
```

```typescript
// app/act/index.json/route.ts
import { actIndexHandler } from '@act/runtime/next';
import { getCurrentUser } from '@/lib/auth';
import { db } from '@/lib/db';

export const GET = actIndexHandler({
  resolveIndex: async (req) => {
    const user = await getCurrentUser(req);
    if (!user) return { unauthorized: true };

    const docs = await db.documents.findMany({
      where: { tenantId: user.tenantId, accessibleTo: user.id },
      select: { id: true, title: true, summary: true, updatedAt: true, parentId: true }
    });

    return {
      nodes: docs.map(d => ({
        id: `doc/${d.id}`,
        type: 'article',
        title: d.title,
        summary: d.summary,
        tokens: { summary: estimateTokens(d.summary), body: 0 },
        etag: hashFor(d, user),
        updated_at: d.updatedAt.toISOString(),
        parent: d.parentId ? `doc/${d.parentId}` : null,
        children: []
      }))
    };
  }
});
```

```typescript
// app/act/n/[id]/route.ts
import { actNodeHandler } from '@act/runtime/next';
import { getCurrentUser } from '@/lib/auth';
import { db } from '@/lib/db';

export const GET = actNodeHandler({
  resolveNode: async (req, { id }) => {
    const user = await getCurrentUser(req);
    if (!user) return { unauthorized: true };

    const docId = id.replace(/^doc\//, '');
    const doc = await db.documents.findFirst({
      where: { id: docId, tenantId: user.tenantId, accessibleTo: user.id }
    });
    if (!doc) return { notFound: true };

    return {
      id,
      type: 'article',
      title: doc.title,
      summary: doc.summary,
      content: [
        { type: 'prose', format: 'markdown', text: doc.body }
      ],
      metadata: { tags: doc.tags, author: doc.authorName },
      source: { human_url: `https://app.acme.com/docs/${docId}` }
    };
  }
});
```

The SDK handles:
- `act_version` injection
- `etag` calculation from the resolver's output (or honors a passed-in etag)
- `tokens` estimation (configurable tokenizer)
- HTTP cache headers (`Cache-Control: private`, `Vary: Authorization`, `ETag`)
- `If-None-Match` revalidation (returns `304 Not Modified` on match)
- Error responses (`401`, `404`) per §5.13.2

The application code only writes the *content resolution logic*. Everything else is the SDK.

#### 6.6.2 Generic SDK shape

For frameworks without a first-party adapter, the generic SDK exposes a request handler:

```typescript
import { createActRuntime } from '@act/runtime';

const runtime = createActRuntime({
  resolveManifest: async (request) => { /* ... */ },
  resolveIndex: async (request) => { /* ... */ },
  resolveNode: async (request, { id }) => { /* ... */ },
  resolveSubtree: async (request, { id }) => { /* ... */ },
  tokenizer: 'tiktoken-cl100k',
  cacheControl: { default: 'private, max-age=60' }
});

// Use with any WHATWG-fetch-compatible framework
app.use('/.well-known/act.json', (req, res) => runtime.manifest(req).then(send(res)));
app.use('/act/index.json',       (req, res) => runtime.index(req).then(send(res)));
app.use('/act/n/:id',            (req, res) => runtime.node(req, req.params.id).then(send(res)));
app.use('/act/sub/:id',          (req, res) => runtime.subtree(req, req.params.id).then(send(res)));
```

#### 6.6.3 Mixing static and runtime in one project

A single Next.js app might publish a static ACT for `/marketing/*` (built from CMS at deploy time) and a runtime ACT for `/app/*` (resolved per-request). Both configs live side-by-side in the project: `act.config.js` for the static build, and route handlers under `app/.well-known/` and `app/act/` for the runtime side. The top-level manifest can use `mounts` (§5.13.5) to expose both.

## 7. Build integration

### 7.1 The pipeline

```
[ Source adapters ]
  ├── markdown / mdx files
  ├── CMS (Contentful, Sanity, Storyblok, ...)
  ├── i18n message catalogs
  ├── Component trees (React/Vue/Angular) ── via static analysis + render
  └── programmatic
              │
              ▼  (NodeDraft stream)
[ Merger ] ── multi-source resolution per merge_policy
              │
              ▼
[ Normalizer ] ── ID assignment, type inference, summary extraction
              │
              ▼
[ Tokenizer ] ── token counts per level
              │
              ▼
[ Hasher ] ── deterministic ETags
              │
              ▼
[ Emitter ] ── per-locale: index.json, n/{id}.json, sub/{id}.json
              └── .well-known/act.json
```

### 7.2 Reference generator architecture

Pseudocode for a minimal generator:

```javascript
async function buildAct(config) {
  // Phase 1: collect from all adapters in parallel
  const adapters = await Promise.all(
    config.sources.map(s => loadAdapter(s.type).init(s, ctx))
  );
  const drafts = [];
  for (const adapter of adapters) {
    for await (const draft of adapter.collect()) drafts.push(draft);
  }

  // Phase 2: merge by ID per policy
  const merged = mergeDrafts(drafts, config.merge_policy);

  // Phase 3: normalize (assign IDs, infer types, extract summaries)
  const nodes = merged.map(d => normalize(d, config));

  // Phase 4: build per-locale trees
  const byLocale = groupBy(nodes, n => n.locale ?? config.locales.default);

  for (const [locale, localeNodes] of byLocale) {
    const tree = buildTree(localeNodes);
    for (const node of tree) {
      node.tokens = estimateTokens(node, config.tokenizer);
      node.etag = computeEtag(node);
      writeFile(localePath(locale, `n/${node.id}.json`), node);
    }
    writeFile(localePath(locale, 'index.json'), buildIndex(tree));
    if (config.subtree_bundles) emitSubtrees(tree, locale);
  }

  // Phase 5: manifest at well-known location
  writeFile(config.well_known_path, buildManifest(config, byLocale));
}
```

### 7.3 SPA-specific pipelines

For React/Vue/Angular apps, the component-tree adapter performs additional steps:

**Static-analysis pass** (compile-time):

```
[ Babel/SWC plugin ]
  ├── walk all source files
  ├── identify components with .act field, @actBlock decorator, or useActBlock call
  ├── inline the extract function into a generated registry
  └── emit  .act-registry.js  used by the render-time pass
```

**Render-time pass** (runs during build, after static):

```
[ Headless render runner ]
  ├── for each locale L in config.locales.available:
  │     for each route R discovered from the framework's manifest:
  │       resolve route data (CMS fetches, generateStaticParams, etc.)
  │       render <ActProvider locale=L route=R> <App/> </ActProvider> via SSR
  │       collect block contracts as components mount
  │       yield NodeDraft{ id: R, locale: L, content: blocks, ... }
  └── stream NodeDrafts back to the merger
```

The render-time pass piggybacks on the SSR machinery the framework already has (Next.js `app/` directory, Remix loaders, Angular Universal, Nuxt). It does NOT require additional infrastructure beyond the existing build.

## 9. Comparison to alternatives

ACT now ships in two profiles. The table separates them so the trade-offs are clear.

| Capability | ACT (static) | ACT (runtime) | llms.txt | llms-full.txt | MCP | NLWeb | Schema.org |
|---|---|---|---|---|---|---|---|
| Build-time generation | ✅ | n/a | ✅ | ✅ | n/a | n/a | ✅ |
| Runtime per-request generation | n/a | ✅ | ❌ | ❌ | ✅ | ✅ | ❌ |
| Authenticated / per-user content | ❌ | ✅ | ❌ | ❌ | ✅ | partial | ❌ |
| Full content included | ✅ | ✅ | ❌ | ✅ | ✅ | ✅ | partial |
| Document hierarchy | ✅ | ✅ | flat | ❌ | ✅ | ✅ | ❌ |
| Progressive disclosure | ✅ | ✅ | partial | ❌ | ✅ | ✅ | ❌ |
| Stable IDs / caching | ✅ | ✅ | ❌ | ❌ | partial | ❌ | ❌ |
| Token budget hints | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ | ❌ |
| Multi-locale, first class | ✅ | ✅ | ❌ | ❌ | partial | partial | ✅ |
| Component-driven extraction | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ | ❌ |
| Tool calling / actions | ❌ | ❌ | ❌ | ❌ | ✅ | partial | ❌ |
| Push / subscriptions | ❌ | ❌ | ❌ | ❌ | ✅ | ❌ | ❌ |
| Transport | HTTP files | HTTP endpoints | HTTP file | HTTP file | JSON-RPC over stdio/SSE/HTTP | HTTP+vector | HTML embedded |
| Browser-agent friction | ✅ none | ✅ uses session | ✅ none | ✅ none | medium (connection setup) | medium | ✅ none |

**ACT and MCP are designed to coexist.** A typical SaaS deployment:

- Marketing/docs/blog → **static ACT**
- Authenticated app reads → **runtime ACT**
- Authenticated app actions → **MCP tools**
- Real-time updates → **MCP notifications**

ACT is meant to **complement** the other entries above:

- Reference ACT from `llms.txt` so existing crawlers can find it.
- Wrap an ACT tree as MCP resources for agents that prefer the JSON-RPC protocol.
- Mirror Schema.org types in `metadata.schema_org_type` for downstream consumers.
- Use NLWeb on top of ACT data for query-time conversational interfaces.

## 10. Open questions

These are unresolved and worth thinking through before v1.0:

1. **Tokenization standard.** Whose tokenizer? Approximations diverge ~15% across vendors. Current resolution: declare which tokenizer was used (`tokens.tokenizer: "approx"` | `"cl100k"` | etc.) and let agents discount accordingly. Open: should there be a single recommended default?

2. **Versioned docs.** A site with `/v1/`, `/v2/` versions — separate trees? A `version` field on nodes? Likely: separate manifests under `/v1/.well-known/act.json` etc., with cross-version `supersedes` references.

3. **Streaming updates.** Could nodes carry a `change_feed_url` so agents subscribe to deltas? Useful for newsrooms and high-velocity docs. Defer to v0.2.

4. **Signed manifests.** Should publishers be able to sign their ACT files (e.g., via LLMFeed-style Ed25519) for provenance? Worth aligning with whatever wins in the trust layer; defer.

5. **Component contract evaluation safety.** The `extract` functions in component contracts run at build time. Should they be sandboxed? If a contract throws or returns malformed data, what happens? Recommendation: treat extraction failures as build warnings (not errors), emit an empty block with `metadata.extraction_status: "failed"`, and log clearly.

6. **A/B variant identity.** When `variants: 'all'` is declared, how should variant IDs be generated? Current proposal: `{base_id}@{variant_key}`, e.g. `pricing@control`, `pricing@experiment-a`. Needs convention finalization.

7. **CMS schema mapping standardization.** Each CMS adapter currently defines its own schema mapping shape. Should there be a unified mapping DSL across adapters (similar to how GraphQL Codegen unifies across schema sources)? Likely yes, but defer to v0.2 once we have signal from real adapter implementations.

8. **Cycle handling in `related`.** Cycles allowed in `related`, never in `children`. Spec language to be tightened.

9. **Maximum sizes.** Should there be hard limits on individual node body size? Recommendation only: split nodes that exceed ~10K tokens into children.

10. **Generator-side LLM use.** Is it kosher to use an LLM at build time to write summaries when authors didn't? Yes, but mark it: `summary_source: "llm"` vs `"author"` vs `"extracted"`, so consumers can weight accordingly.

11. **Personalization beyond A/B.** Truly personalized sites (per-user content) can't ship a static ACT. Runtime ACT (§5.13) handles this — the per-user manifest/index/node responses are computed per-request, scoped to the authenticated user. Static ACT is for the canonical/default experience; runtime ACT is for per-identity content.

12. **Component extraction in non-SSR-able apps.** Headless rendering via jsdom/Puppeteer works but is slow and brittle. Is this an acceptable v0.1 fallback or should we require SSR-compatibility?

13. **Embedded structured data.** Is the `data` block sufficient for tabular/graph content, or do we need a first-class `Dataset` node type? Probably first-class for catalogs, datasets, and APIs.

14. **Robots/access control.** Should ACT have a per-node "agents only" or "no train" flag, or strictly defer to robots.txt + Content-Signal headers? Strictly defer — out of scope.

15. **Runtime change feeds.** v0.1 runtime is request/response only. Should v0.2 add an SSE-based change feed (`/act/changes` streaming `{id, etag, deleted?}` events) so long-lived agents can stay in sync without polling? Defer until adoption signal exists.

16. **Runtime ETag determinism across replicas.** If a SaaS is sharded across replicas, the same content should produce the same ETag from any replica. Recommendation: derive ETags from `hash(content_payload + auth_identity)` deterministically, never from request-local timing or random nonces. Document explicitly.

17. **Discovery for runtime-only deployments.** A pure-SaaS app behind auth has no public `/.well-known/act.json` to discover. Should the protocol define a discovery hand-off (e.g., the app's HTML emits `<link rel="act" href="/.well-known/act.json" type="runtime">` in authenticated contexts)? Likely yes. The exact mechanism — meta tag, HTTP header, both — needs convention.

18. **Runtime + MCP unified server.** Should there be a reference implementation that exposes the same content tree via both ACT runtime endpoints and MCP resources, so SaaS authors get both with one set of resolver functions? Strong yes for adoption; defer to a separate package, not the spec.

## Appendix A — Why "ACT"?

Agent Content Tree. Three letters, pronounceable, doesn't collide with widely-used terms in the LLM/web space, and the verb meaning ("to act") is a fine fit for the agentic web framing. Files are `.act.json`. Generators are `act-*`. The verb form lets you say "the site ACTs" without it being painful.

If a name conflict surfaces during community review, fallback names: ACG (Agent Content Graph), AGTREE.

## Appendix B — Adoption strategy

For ACT to become useful rather than just exist, the strategy splits along two fronts:

**Front 1: docs/content sites (the easy adoption)**

1. Ship reference generators for **Astro** and **Hugo** first. That's where the docs/blog mass lives, and the markdown adapter is trivial.
2. Land it as a default in **Docusaurus** and **MkDocs** so existing docs sites get ACT for free.
3. Get **llmstxt.org** to acknowledge ACT as the structured-content companion to `llms.txt`.

**Front 2: corporate/marketing/app sites (the hard adoption, but the bigger prize)**

4. Ship a **Next.js** plugin with the `react-component` adapter — this is where most modern marketing sites live.
5. Ship **Contentful**, **Sanity**, and **Storyblok** adapter packages on day one. Without first-party CMS support, marketing sites won't try ACT.
6. Partner with one **design system** (e.g. shadcn/ui, Material UI, Chakra) to ship `.act` contracts on their default components. Suddenly any site using that design system gets ACT for free.
7. Provide a **Vercel/Netlify build integration** so the manifest auto-deploys to `/.well-known/`.

**Front 3: B2B SaaS apps (runtime profile)**

8. Ship a **runtime SDK** for Next.js, Express, FastAPI, and a generic WHATWG-fetch handler. The pattern is small (manifest/index/node/subtree resolvers); the SDK should make it ~30 lines of code to expose ACT from an existing app.
9. Ship a **paired ACT + MCP server** reference implementation so a SaaS can expose both with one set of resolvers — ACT for reads, MCP tools for writes. Most SaaS authors want both, and bundling them lowers the adoption tax dramatically.
10. Land first-party adapters in **Notion-style document platforms**, **CRM/ticketing tools**, and **collaboration suites**. These are the use cases where browser-based agents are most painful today.

**Cross-cutting:**

11. **Get one major LLM client to actually look for `/.well-known/act.json`.** Even one is enough to create demand.
12. **Publish a validator** at e.g. `act-spec.org/validate` so authors can sanity-check their feed.
13. **Document the "wrap me as MCP" path explicitly.** Anyone running an MCP server should be able to expose an ACT tree in ~50 lines of code.
14. **Keep the core spec painfully minimal.** Adapters, instrumentation, and the runtime SDK can grow; the wire format must not.

## Appendix C — Comparison: a 10K-node site

For a hypothetical 10,000-node documentation site with average node size ~500 tokens:

| Approach | Initial fetch | Targeted fetch | Per-update cost |
|---|---|---|---|
| Scrape HTML | ~25M tokens | full re-scrape | full re-scrape |
| `llms-full.txt` | ~5M tokens | ~5M tokens (no granularity) | full re-fetch |
| ACT (index + 5 needed nodes) | ~150K tokens (index) + ~2.5K (5 nodes) | ~2.5K tokens | per-node ETag check |

The asymmetry is the point: ACT pays a fixed cost to give the agent a map, then lets the agent fetch only what's needed.

## Appendix D — Why this matters for component-driven sites specifically

A 50-page corporate marketing site is, in token terms, much smaller than a 10K-node docs site — but it's much *harder* for agents to consume today. Marketing pages are JS-heavy, content-light per page, image-heavy, and locked behind client-rendered React with content composed from CMS at request time. A naive scrape of `acme.com/pricing` returns a hydration shell with very little actual copy.

ACT changes this asymmetry. With a properly instrumented design system, the same page emits a clean ~300-token structured block list at build time. Agents reading from ACT see the same pricing tiers, FAQ items, and CTAs that a human sees — without scraping HTML, executing JavaScript, or guessing at structure.

The corporate sites that win in the agentic web aren't the ones with the prettiest landing pages. They're the ones whose landing pages an AI assistant can actually *understand* when a customer asks "what does Acme cost?"

## Appendix E — ACT and MCP: the relationship, plainly

The most common confusion about ACT runtime is "isn't this just MCP?" The answer is no, but the boundary is worth stating directly.

**MCP** is a *protocol* — JSON-RPC over stdio, SSE, or HTTP — for agents to interact with servers. Its core verbs are `list_resources`, `read_resource`, `call_tool`, and a notification stream. It assumes a connection lifecycle (initialize → use → close), and it supports both reads and writes (tools).

**ACT** is a *content shape* — a JSON document tree with progressive disclosure, stable IDs, token estimates, and typed blocks. It can be delivered statically (files on a CDN) or dynamically (HTTP endpoints with auth). It is read-only.

The genuine overlap is "structured read-only content." For that case, you could expose your content as MCP resources or as ACT — both work. The decision is about transport ergonomics:

- **HTTP-native**: ACT runtime is "just an API." Any client that can `fetch()` can consume it. Browser-based agents using user sessions are trivially supported. Standard CDNs, proxies, and caches Just Work. No connection setup.
- **Protocol-native**: MCP has a connection lifecycle, OAuth setup, subscription support, and a unified surface for tools alongside resources. Tooling is growing; first-party clients exist.

For *anything that mutates state, takes actions, or pushes updates*, MCP is correct and ACT is silent. ACT does not define tools, mutations, or notifications, and it shouldn't — that's a different job.

The intended pattern for a mature SaaS:

```
┌─────────────────────────────┐
│   Marketing site, blog,     │
│   public docs, catalog      │  ──► Static ACT (CDN)
└─────────────────────────────┘

┌─────────────────────────────┐
│   Authenticated app reads   │  ──► Runtime ACT (HTTP + session)
│   (per-user, per-tenant)    │
└─────────────────────────────┘

┌─────────────────────────────┐
│   Authenticated app writes, │  ──► MCP (tools + notifications)
│   actions, push updates     │
└─────────────────────────────┘
```

All three speak shapes that share lineage. A reference implementation that exposes the same content tree via both ACT runtime endpoints and MCP resources — from one set of resolver functions — is on the v0.1 adoption roadmap (Appendix B, item 9). Authors should not have to pick; they should get both.

The short version: **ACT is where the content lives. MCP is where the verbs live. Most SaaS apps want both.**
