# PRD-104 — Internationalization (locale model, cross-locale references, untranslated keys)

## Status

`Accepted`

---

## Engineering preamble

The preamble is for humans deciding whether this PRD belongs in the work queue. It is non-normative. The Specification section below is the contract.

### Problem

The v0.1 working draft (`docs/plan/v0.1-draft.md` §5.12) treats internationalization as a first-class concern but leaves three concrete mechanics unspecified at a level adapters and consumers can both implement against:

1. **Locale model in the manifest.** §5.12.1 sketches a `locales` block with `default`, `available`, and `manifest_url_template`. It does not specify the BCP-47 grammar producers MUST emit, the rule that `default` MUST appear in `available`, the legal omission case (single-locale Plus producers), or the relationship between locale-prefixed IDs and per-locale manifest URL templates as alternative deployment patterns.
2. **Cross-locale references.** §5.12.2 shows a `translation_of` field at the top level of a node. It does not say whether this composes with the `related` cross-reference graph that sibling content-block PRDs own, whether a denser `metadata.translations` array form is permitted, or whether the convention is symmetric (en→de AND de→en).
3. **Untranslated keys.** §5.12.4 introduces a `metadata.translation_status: "fallback"` marker and a fallback-from value. It does not enumerate the closed set of statuses, does not forbid `null` on required string fields, and does not say what producers do for whole-node-missing translations as opposed to block-level partials.

The corporate marketing example (draft §6.5, §8.4) is the canonical Plus i18n deployment that exercises every one of these — Next.js + Contentful + `next-intl` across four locales, with cross-locale references between `/pricing` (en-US) and `/precios` (es-ES). Until this PRD lands, the i18n adapter (PRD-207, P2) cannot define its emission contract, generators cannot validate per-locale output, and PRD-107's pre-pinned rule that "i18n manifest is Plus" (R10) lacks the precision it explicitly delegates to PRD-104.

This is a Tier-A foundational concern wearing a Tier-E label: i18n is not in `000-gaps-and-resolutions.md` as an A/B/C/D gap, but it touches manifest, index, and node shapes that PRD-100 / PRD-102 are pinning simultaneously, and PRD-207 cannot leave Draft until PRD-104 defines what it produces.

### Goals

1. Define the manifest `locales` block precisely (`default`, `available`, optional `manifest_url_template`), including a v0.1-friendly subset of BCP-47.
2. Specify two interoperable per-locale layout patterns (locale-prefixed IDs vs per-locale manifests) and the rules that govern picking one.
3. Specify the per-node locale field (`metadata.locale`) and when it is REQUIRED vs MAY-be-omitted.
4. Specify cross-locale references via both the `related` cross-reference graph and the denser `metadata.translations` array, and require consumers to tolerate both.
5. Pin the closed enum for `metadata.translation_status` and the contract that fallback content MUST be emitted (no `null`).
6. Restate the locale variant convention as an extension of the variant convention owned by sibling PRDs, NOT a redefinition of it.
7. Provide JSON Schema fragments for the locales block and the locale-related node metadata, plus positive and negative fixtures covering every MUST.

### Non-goals

1. **The manifest envelope itself.** Owned by the wire-format / manifest-shape sibling PRD (PRD-100). PRD-104 only defines the `locales` block within that envelope.
2. **The node envelope, content blocks, or the `related` shape.** Owned by sibling PRDs (PRD-100 / PRD-102). PRD-104 defines the locale-specific fields and cites the `related.relation` value `"translation_of"` as a reserved relation.
3. **The variant convention itself.** Owned by sibling PRD covering content blocks / variant identity (gap D2 — `{base_id}@{variant_key}`). PRD-104 only declares that `source: "locale"` is a permitted variant source and that locale-derived `variant_key`s use the BCP-47 subset.
4. **The i18n source adapter.** Owned by PRD-207 (P2). PRD-104 defines the wire-format contract that PRD-207's output MUST satisfy.
5. **The search envelope.** Owned by sibling PRD (PRD-100 / PRD-106). PRD-104 only specifies the convention that `search_url_template` SHOULD accept a `locale` parameter.
6. **Authentication / per-tenant scoping interaction.** Owned by PRD-109 sibling. PRD-104 cites the rule that locale-prefixed IDs MUST NOT bypass per-tenant scoping; the implementation lives in PRD-109.
7. **Full BCP-47 grammar.** v0.1 ships a deliberate subset (see Specification). Full BCP-47 — extlangs, multi-variant tags, private-use, grandfathered — is deferred to v0.2+ and is non-blocking because the relaxation from a tighter regex to BCP-47-strict is MAJOR per PRD-108 R5(6) and would land at a MAJOR boundary anyway.

1. Should the `metadata.translations` dense form REQUIRE listing every locale in `locales.available`, or only those that actually exist? ## Specification

This is the normative section. Everything below uses RFC 2119 keywords (MUST, MUST NOT, SHOULD, SHOULD NOT, MAY) where requirements are imposed. Lowercase "must" and "should" are non-normative prose.

### Conformance level

i18n is a **Plus**-tier capability per PRD-107-R10. Most PRD-104 requirements apply at Plus. A small subset (the variant-convention extension, the `metadata.locale` shape) applies wherever a producer ships locale-aware content at all, including Standard producers that emit a single locale with `metadata.locale` for forward compatibility.

- **Core:** No additive requirements. Core producers MAY ship single-locale content; locales are not Core.
- **Standard:** No additive requirements. Cross-locale references via `related.relation: "translation_of"` use the `related` shape owned by the sibling content-block PRD, which is Standard-tier; the `relation` value `"translation_of"` is reserved by PRD-104 (PRD-104-R8) for use across all levels.
- **Plus:** PRD-104-R1 through PRD-104-R14 (the manifest `locales` block, per-node `metadata.locale` for Pattern 1, cross-locale references, `metadata.translation_status`, per-locale URL templates, single-locale-omission rule).

### Sibling-PRD assumptions

PRD-104 is being authored in parallel with the manifest-and-node sibling PRDs. The following assumptions are made explicitly so BDFL review can validate consistency:

1. **Manifest envelope.** The manifest is a top-level JSON object with sibling fields including at least `act_version`, `site`, `index_url`, `node_url_template`. The optional `locales` block is a top-level sibling to those fields, per draft §5.12.1.
2. **`site.locale`.** The manifest's `site` object MAY carry a `locale` field (BCP-47 string) declaring the locale of THIS manifest. In Pattern 2 (per-locale manifests), each per-locale manifest carries its own `site.locale`. In Pattern 1 (single manifest, locale-prefixed IDs), `site.locale` SHOULD equal `locales.default`.
3. **Node envelope and `metadata`.** Nodes are JSON objects; node-level metadata lives under a `metadata` sub-object. PRD-104 places `locale`, `translation_status`, `fallback_from`, and `translations` under `metadata`. The v0.1 draft §5.5 lines 666–667 sketches a top-level `locale` and top-level `translation_of`; PRD-104 supersedes that sketch by placing those fields under `metadata` and inside `related` respectively. **If sibling PRD-100 ships top-level `locale` / `translation_of` normatively instead, PRD-104 MUST be amended; this is an explicit conflict flag.**
4. **`related` shape.** PRD-104 assumes the `related` cross-reference graph (sibling content-block PRD) is an array whose items support a discriminating `relation` field. The form `[{ "id": "...", "relation": "translation_of", "locale": "..." }]` is what PRD-104-R8 emits. **If sibling PRD ships `related` as a plain array of ID strings, the `metadata.translations` form (PRD-104-R9) becomes the primary carrier and PRD-104-R8 is downgraded to MAY.** Conflict flag.
5. **Variant convention.** PRD-104 assumes the variant convention `{base_id}@{variant_key}` (gap D2) is owned by sibling PRD and is permissive enough to accept `variant_key` values that are BCP-47 subset locales (e.g., `pricing@es-es`). PRD-104 only adds `source: "locale"` as a permitted value of `metadata.variant.source`.
6. **Capability flags.** The capability flag set (`capabilities.etag`, `capabilities.subtree`, `capabilities.search.template_advertised`, `capabilities.ndjson_index`) is owned by sibling manifest PRD. PRD-104 does NOT introduce a new `capabilities.i18n` flag; the presence of `locales` in the manifest is the signal.
7. **Search URL template.** The sibling PRD owning `search_url_template` is responsible for the substitution grammar; PRD-104 only states the convention that a `locale` parameter SHOULD be accepted.

### Normative requirements

#### The manifest `locales` block

**PRD-104-R1.** *(Plus)* When a producer's content includes more than one locale, the manifest MUST include a `locales` object at the top level of the manifest envelope. When a producer's content is single-locale, the manifest MAY omit `locales` (see PRD-104-R14).

**PRD-104-R2.** *(Plus)* When the `locales` object is present, it MUST include `default` (a BCP-47 locale string matching the v0.1 subset regex `^[a-z]{2,3}(-[A-Z][a-z]{3})?(-[A-Z]{2})?$`) and `available` (a non-empty array of unique BCP-47 locale strings, each matching the same regex).

**PRD-104-R3.** *(Plus)* The value of `locales.default` MUST appear in `locales.available`. Producers that violate this rule produce an invalid manifest; consumers MUST reject such manifests as malformed (manifest validation error per PRD-100, surfaced as a `gaps` entry by PRD-600 citing PRD-104-R3).

**PRD-104-R4.** *(Plus)* The `locales` object MAY include `manifest_url_template`, a URL template containing the literal `{locale}` placeholder. When present, `manifest_url_template` advertises Pattern 2 (per-locale manifests). When absent, Pattern 1 (locale-prefixed IDs in a single manifest) is implied.

#### Per-locale data layout: Pattern 1 and Pattern 2

**PRD-104-R5.** *(Plus)* **Pattern 1 — locale-prefixed IDs.** Producers MAY ship a single manifest whose `index_url` and `node_url_template` cover all locales, with node IDs that include a locale segment (e.g., `en/intro/getting-started`, `de/intro/getting-started`). Pattern 1 MUST satisfy:

- The ID grammar (gap A3, owned by PRD-100 sibling) accepts `/` as a path separator and lowercase ASCII alphanumerics; locale-prefixed IDs MUST conform.
- Every node MUST carry `metadata.locale` declaring its locale (a BCP-47 subset string per the regex in PRD-104-R2).
- The locale segment in the ID and the `metadata.locale` value MUST agree (e.g., an `id` of `en/pricing` MUST carry `metadata.locale: "en"` or `metadata.locale: "en-US"` where `en` is the lowercased primary subtag of `en-US`).

**PRD-104-R6.** *(Plus)* **Pattern 2 — per-locale manifests.** When `locales.manifest_url_template` is present, each entry of `locales.available` MUST resolve to a per-locale manifest at the URL produced by substituting that locale into the template. Each per-locale manifest:

- MUST declare its locale via `site.locale` (the BCP-47 subset string).
- MAY declare its own `locales` block; if it does, `locales.default` and `locales.available` SHOULD be consistent with the parent manifest's. If the per-locale manifest omits `locales`, consumers MUST infer it from the parent manifest.
- Has its own `index_url` and `node_url_template`. Node IDs in per-locale manifests are locale-bare (Pattern 2's whole point is that the per-locale manifest's URL prefix carries the locale identity).
- Nodes in Pattern 2 MAY omit `metadata.locale`; the per-locale manifest's `site.locale` is authoritative.

**PRD-104-R7.** *(Plus)* Producers SHOULD pick exactly one of Pattern 1 or Pattern 2 per site. Mixing patterns within a single site (some locales as prefixed IDs, others as separate manifests) is permitted but discouraged; PRD-600 emits a warning. A multi-locale site that ships locale-bare IDs WITHOUT either Pattern 1's locale prefix OR Pattern 2's per-locale manifest MUST be rejected as ambiguous.

#### Cross-locale references

**PRD-104-R8.** *(Standard, by sibling-PRD inheritance — the `related` shape is Standard, the `relation` value `"translation_of"` is reserved by PRD-104 at all levels.)* When a node is a translation of another node, producers MAY emit a `related` entry with `relation: "translation_of"` carrying the source-language node's `id` and `locale`. Bidirectional emission is RECOMMENDED: en→de via `translation_of` AND de→en via `translation_of` (the relation is symmetric for cross-locale-translation purposes; producers MAY use a directed `translated_into` relation if the sibling `related` PRD reserves it, but PRD-104 does not require directionality).

**PRD-104-R9.** *(Plus)* Producers MAY emit a denser alternative form, `metadata.translations`, as an array of `{ locale, id }` objects listing every other locale in which a translation of this node exists. PRD-104-R8 and PRD-104-R9 are not mutually exclusive — producers MAY emit both, and consumers MUST tolerate either or both. When both are emitted, they MUST agree (same locale-to-id mappings); a discrepancy is a validator finding, not a hard error.

#### Untranslated keys

**PRD-104-R10.** *(Plus)* When a content key (e.g., a `marketing:hero` block's `headline` field, or a `prose` block's `text` field) lacks a translation in the requested locale, the producer SHOULD fall back to the default-locale value AND set `metadata.translation_status: "fallback"` on the affected block. When a whole node lacks a translation in the requested locale, the producer SHOULD set `metadata.translation_status: "fallback"` on the node and substitute default-locale content. The `metadata.fallback_from` field MUST be set to the locale whose value was substituted (typically `locales.default`).

**PRD-104-R11.** *(Plus)* `metadata.translation_status` is a **closed enum** with values `"complete" | "partial" | "fallback" | "missing"`:

- `"complete"` — every translatable key in this scope is translated to the requested locale.
- `"partial"` — at least one block within this node is `"fallback"` or `"missing"`; the node as a whole is partially translated.
- `"fallback"` — this scope (block or node) is showing default-locale content because the requested-locale translation is unavailable.
- `"missing"` — this scope is acknowledged untranslated, with no fallback substitution. Permitted ONLY at the node level when the whole node has no representation in the requested locale; in that case the node MAY be omitted from the index entirely instead. **`"missing"` MUST NOT appear on a block whose required string fields are `null`** (see PRD-104-R12).

Adding a value to this enum is MAJOR per PRD-108 R5(4).

**PRD-104-R12.** *(Plus)* Producers MUST NOT emit `null` for required string fields on content blocks (e.g., `headline`, `text`, `title`) as a translation-missing signal. The contract is fallback substitution per PRD-104-R10. A block with `translation_status: "missing"` AND `null` values on required string fields MUST be rejected by validators.

**PRD-104-R13.** *(Plus)* A node carrying any block with `translation_status: "fallback"` SHOULD itself carry `metadata.translation_status: "partial"` (or `"fallback"` if every translatable block is fallback content). Consumers MAY infer node-level status from block-level status; producers SHOULD make it explicit at the node level.

#### Single-locale Plus producers

**PRD-104-R14.** *(Plus)* A Plus producer whose content is single-locale MAY omit the `locales` block from the manifest. This is the precise rule PRD-107-R10 delegates to PRD-104:

- A producer MAY omit `locales` when `locales.available` would have exactly one entry equal to `locales.default`.
- A producer that omits `locales` MUST either omit `site.locale` entirely OR set `site.locale` to a single BCP-47 subset string. Consumers receiving a manifest without `locales` and without `site.locale` MUST treat the content as locale-undeclared and proceed without locale-aware behavior.
- A producer that anticipates adding locales soon SHOULD include `locales` with `available: ["<default>"]` even before the second locale ships, to advertise multi-locale intent.

#### Variant convention extension

**PRD-104-R15.** *(Standard / Plus, by sibling-PRD inheritance.)* The variant convention `{base_id}@{variant_key}` (sibling content-block PRD owns; gap D2) is permitted to carry locale-derived variants. When used for locale variants:

- `variant_key` MUST match the BCP-47 subset regex with hyphens lowercased: `^[a-z]{2,3}(-[a-z]{4})?(-[a-z]{2})?$`. Note the lowercase `[a-z]{2}` for region — variant_key is locale-derived but lowercase per gap D2's `[a-z0-9-]+` constraint.
- `metadata.variant.source` MUST be `"locale"` for locale-derived variants.
- The recommended primary path for multi-locale sites is **Pattern 1** (locale-prefixed IDs, PRD-104-R5). The locale-variant convention is the fallback for sites that need to keep the canonical `id` locale-free.
- A node MAY use the variant convention AND simultaneously carry `metadata.locale`; the two MUST agree.

#### Search and i18n

**PRD-104-R16.** *(Plus, informational extension)* When a Plus producer ships both `search_url_template` (sibling PRD) and a multi-locale manifest, the `search_url_template` SHOULD accept a `{locale}` parameter alongside `{query}`. Example: `/act/search?q={query}&locale={locale}`. PRD-104 does not pin the substitution grammar — that is owned by the sibling search PRD. Consumers that omit the locale parameter MUST receive default-locale results.

### Wire format / interface definition

#### `manifest.locales` (JSON Schema fragment)

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://act-spec.org/schemas/104/locales-block.schema.json",
  "title": "ACT Manifest — locales block (PRD-104)",
  "type": "object",
  "required": ["default", "available"],
  "additionalProperties": false,
  "properties": {
    "default": {
      "type": "string",
      "pattern": "^[a-z]{2,3}(-[A-Z][a-z]{3})?(-[A-Z]{2})?$"
    },
    "available": {
      "type": "array",
      "minItems": 1,
      "uniqueItems": true,
      "items": {
        "type": "string",
        "pattern": "^[a-z]{2,3}(-[A-Z][a-z]{3})?(-[A-Z]{2})?$"
      }
    },
    "manifest_url_template": {
      "type": "string",
      "pattern": ".*\\{locale\\}.*"
    }
  }
}
```

The cross-validation rule that `default` MUST appear in `available` (PRD-104-R3) is enforced by PRD-600's validator, not by the schema fragment alone. The full schema is saved at `schemas/104/locales-block.schema.json`.

The BCP-47 subset accepts:

- 2- or 3-letter primary language subtag (lowercase): `en`, `es`, `de`, `ja`, `zh`, `eng`, etc.
- Optional 4-letter script subtag (capitalized): `Hant`, `Hans`, `Latn`.
- Optional 2-letter region subtag (uppercase): `US`, `ES`, `DE`, `JP`, `BR`.

Examples accepted: `en`, `en-US`, `zh-Hant`, `zh-Hant-HK`, `pt-Latn-BR`. Examples rejected (v0.1): `en-US-x-private`, `es-419` (numeric region), `i-klingon` (grandfathered). Producers needing those formats SHOULD wait for v0.2 (a relaxation of this regex is MAJOR per PRD-108 R5(6)).

#### Per-node locale metadata (JSON Schema fragment)

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://act-spec.org/schemas/104/node-locale-metadata.schema.json",
  "title": "ACT Node — locale-related metadata (PRD-104)",
  "type": "object",
  "additionalProperties": true,
  "properties": {
    "metadata": {
      "type": "object",
      "additionalProperties": true,
      "properties": {
        "locale": {
          "type": "string",
          "pattern": "^[a-z]{2,3}(-[A-Z][a-z]{3})?(-[A-Z]{2})?$"
        },
        "translation_status": {
          "type": "string",
          "enum": ["complete", "partial", "fallback", "missing"]
        },
        "fallback_from": {
          "type": "string",
          "pattern": "^[a-z]{2,3}(-[A-Z][a-z]{3})?(-[A-Z]{2})?$"
        },
        "translations": {
          "type": "array",
          "items": {
            "type": "object",
            "required": ["locale", "id"],
            "additionalProperties": false,
            "properties": {
              "locale": {
                "type": "string",
                "pattern": "^[a-z]{2,3}(-[A-Z][a-z]{3})?(-[A-Z]{2})?$"
              },
              "id": { "type": "string" }
            }
          }
        }
      }
    }
  }
}
```

The full schema is saved at `schemas/104/node-locale-metadata.schema.json`. The conditional rule (PRD-104-R12: `translation_status: "missing"` MUST NOT coexist with `null` values on required string fields) is enforced by PRD-600's validator.

### Errors

| Condition | Response | Notes |
|---|---|---|
| `locales.default` is not present in `locales.available` | Manifest validation error (build-time for static; configuration error at startup for runtime) | Per PRD-104-R3. PRD-600 emits a `gaps` entry citing PRD-104-R3. |
| `locales.default` or `locales.available` does not match the BCP-47 subset regex | Manifest validation error | Per PRD-104-R2. PRD-600 emits a `gaps` entry. |
| Multi-locale site emits locale-bare IDs without Pattern 1 prefixing or Pattern 2 per-locale manifests | Manifest / index validation error | Per PRD-104-R7. PRD-600 emits a `gaps` entry. |
| Block carries `translation_status: "missing"` AND `null` on a required string field | Build error (static) / 5xx-then-fix (runtime, since this is producer bug not request error) | Per PRD-104-R12. PRD-600 rejects on validation. |
| Pattern 1 node carries an `id` whose locale prefix disagrees with `metadata.locale` | Validation error | Per PRD-104-R5 third bullet. |
| Pattern 2 per-locale manifest does not resolve at the templated URL | Discovery error; consumer treats the locale as unavailable | Per PRD-104-R6. PRD-600 emits a `gaps` entry probing each `available` entry. |
| Producer mixes Pattern 1 and Pattern 2 within a single site | Warning, not error | Per PRD-104-R7. Permitted but discouraged. |
| Both `related.relation: "translation_of"` and `metadata.translations` are emitted with disagreeing data | Validator finding (warning), not error | Per PRD-104-R9. Producer SHOULD reconcile. |

ACT runtime endpoints follow the error envelope defined by sibling PRD-100 / PRD-106 (gap A4). PRD-104 does not introduce new HTTP status codes.

## Versioning & compatibility

Per PRD-108, classify each kind of change to PRD-104 as MAJOR or MINOR.

| Kind of change | MAJOR/MINOR | Notes |
|---|---|---|
| Add an optional field inside the `locales` block (e.g., `locales.fallback_chain`) | MINOR | Per PRD-108 R4(1). |
| Add a documented BCP-47 escape hatch (e.g., a separate `locales.extended` array accepting full BCP-47) | MINOR | Adds an optional capability without obligating producers. |
| Relax the BCP-47 subset regex to full BCP-47 (changing the pattern) | MAJOR | Per PRD-108 R5(6). Existing strict-subset producers remain valid; existing consumers that hard-wired the regex would need to update. |
| Add a value to the `metadata.translation_status` closed enum (e.g., `"machine"`) | MAJOR | Per PRD-108 R5(4). |
| Tighten a SHOULD to a MUST (e.g., promote PRD-104-R8 from MAY to MUST) | MAJOR | Per PRD-108 R5(3). |
| Add a new optional `metadata.*` field (e.g., `metadata.translation_method`) | MINOR | Per PRD-108 R4(1). |
| Promote `locales` from Plus to Standard | MAJOR | Affects PRD-107's level matrix; coordinate with PRD-107 amendment. |
| Change which Pattern (1 or 2) is recommended | MINOR | The recommendation is a SHOULD on producer guidance, not a normative MUST/MUST NOT shift. |
| Forbid Pattern-mixing (tighten PRD-104-R7's "discouraged" to "MUST NOT mix") | MAJOR | Per PRD-108 R5(3). |
| Specify the `search_url_template` `{locale}` substitution grammar normatively | MAJOR or MINOR depending on whether substitution was already implicitly required | Coordinate with sibling search PRD; PRD-104 alone cannot pin this. |

### Forward compatibility

Per PRD-108, consumers MUST tolerate unknown optional fields in the `locales` block, in `metadata.*`, and in `related` entries. A consumer encountering an unrecognized `metadata.translation_status` value MUST treat the manifest's `act_version` as authoritative — if the consumer's MAJOR matches and the value is unrecognized, the consumer SHOULD log a warning and treat the status as `"complete"` (the safest default, equivalent to ignoring the field). If MAJOR mismatches, PRD-108 R8 governs.

### Backward compatibility

- A producer that adds a new locale to `locales.available` is making a producer-side data change, not a spec change; no version bump is required.
- A producer that switches from Pattern 1 to Pattern 2 (or vice versa) is making a deployment-shape change visible to consumers; the manifest's `act_version` need not bump but the `etag` of the manifest will change, triggering normal revalidation.
- A producer migrating from the v0.1 draft sketch (top-level `locale` and top-level `translation_of`) to PRD-104's normative shape (`metadata.locale`, `related.relation: "translation_of"` plus `metadata.translations`) MUST ship the change in a MINOR-bumped manifest if PRD-104's shape becomes a MUST. Until then, consumers SHOULD tolerate the draft shape as a transitional convenience.

## Security considerations

PRD-109 (sibling) governs the project-wide security posture. PRD-104 documents only the deltas i18n introduces.

- **Per-locale endpoints don't leak existence in new ways.** A 404 on `/de-DE/act/n/preise.json` carries the same semantics as a 404 on a default-locale node: the resource is absent or not accessible to the requester. Cite PRD-109 for the runtime 404-vs-403 disambiguation rule (gap A4); PRD-104 inherits it without change.
- **`Accept-Language` is informational only.** Consumers and producers MUST NOT use the `Accept-Language` HTTP header for authentication or authorization scoping. The header is a hint; per-tenant scoping (gap C4 + PRD-109) takes precedence. A producer that returns different content for the same node ID + identity based solely on `Accept-Language` is performing locale negotiation, not auth scoping; the response's `metadata.locale` and ETag MUST reflect the negotiated locale.
- **Locale-prefixed IDs MUST follow the ID grammar.** Pattern 1 IDs include path segments such as `en-us/pricing`. Each segment MUST conform to the ID grammar (gap A3, owned by PRD-100 sibling). A locale prefix MUST NOT be permitted to include characters the grammar disallows; producers that lowercase locale tags to fit (e.g., `EN-US` → `en-us`) MUST do so consistently.
- **Locale prefixes MUST NOT bypass per-tenant scoping.** A runtime ACT server serving Pattern 1 with locale-prefixed IDs MUST apply the same per-tenant authorization checks (PRD-109) that it applies to locale-bare IDs. Locale prefixing is a content-organization mechanism, not a security boundary. A consumer that is authorized for `pricing` but not for `internal/pricing-internal` MUST be similarly authorized for `en-us/pricing` but not `en-us/internal/pricing-internal`.
- **Cross-locale references SHOULD NOT leak existence across tenants.** When a node carries `metadata.translations`, the listed locale-and-id pairs MUST refer to nodes the requester can access. Producers MUST filter the array to translation pairs the requester is authorized to see; emitting a `translations: [{ locale: "en-US", id: "tenant-a/internal/foo" }]` to a tenant-B requester leaks tenant-A's existence.
- **Fallback content MUST NOT cross tenancy boundaries.** When PRD-104-R10 substitutes default-locale content for a missing translation, the substituted content MUST come from a node the requester is authorized to see. If the default-locale node is itself unauthorized, the producer MUST emit `translation_status: "missing"` (per PRD-104-R11) and either omit the node from the response OR substitute a placeholder per PRD-109's content-redaction rules.
- **No new DoS surface.** The locales block adds bounded structure (one default, one array, one optional template); no recursion, no unbounded growth. PRD-100's manifest size limits apply unchanged.
