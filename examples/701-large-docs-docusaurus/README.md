# PRD-701 — Tinybox SDK docs (Docusaurus large docs)

Reference example for [PRD-701](../../prd/701-large-docs-docusaurus.md): a
Docusaurus 3.x documentation site (200-500 ACT nodes, 4-level sidebar
hierarchy) that validates clean against PRD-600 with `achieved.level ===
'standard'`.

The site composes:

- `@act-spec/docusaurus` (PRD-404) — the generator plugin under test.
- `@act-spec/markdown-adapter` (PRD-201) — auto-wired by PRD-404-R5
  against the procedurally generated `docs/` corpus.
- `@act-spec/generator-core` — the PRD-400 pipeline.
- `@act-spec/validator` (PRD-600) — the conformance gate.

The corpus is regenerated by `scripts/generate-corpus.ts` so the example
is reproducible byte-for-byte. The fixed top of the tree (intro,
getting-started, concepts, api, recipes, troubleshooting, changelog)
mirrors PRD-701-R5; the `recipes/` and `troubleshooting/` branches expand
procedurally to hit PRD-701-R3's 200-500 envelope.

## Running locally

```sh
pnpm -F @act-spec/example-701-large-docs-docusaurus generate-corpus
pnpm -F @act-spec/example-701-large-docs-docusaurus build
pnpm -F @act-spec/example-701-large-docs-docusaurus validate
# Or in one command:
pnpm -F @act-spec/example-701-large-docs-docusaurus conformance
```

The build emits the ACT artifact set:

```
build/
├── .well-known/act.json          — manifest declaring conformance: standard
├── act/
│   ├── index.json                — one entry per emitted node
│   ├── nodes/<id>.json           — one node file per index entry
│   └── subtrees/<id>.json        — one subtree file per sidebar category
└── .act-build-report.json        — sidecar build report (PRD-404-R12)
```

The validator script reads the artifact set, runs PRD-600's static walk,
and asserts:

- `gaps.length === 0` (PRD-701-R12 / R16).
- `declared.level === 'standard'` (PRD-701-R13).
- `achieved.level === 'standard'` (PRD-701-R12).
- `delivery === 'static'` (PRD-701-R10).
- `subtreeFiles.length > 0` (PRD-701-R6).
- Corpus envelope: 200 ≤ nodes ≤ 500 (PRD-701-R3).
- Index ↔ node-files-on-disk consistency (PRD-701-R12 last clause).

## Implementation notes

The example's `scripts/build.ts` exercises the plugin's pipeline
programmatically (mirrors `packages/docusaurus/conformance.ts`) rather
than spawning `docusaurus build`. This is faithful to PRD-701-R10's
contract — which is over the **ACT-owned** paths under `build/` — and
sidesteps the heavy `@docusaurus/preset-classic` + theme + React install
footprint the full Docusaurus CLI requires.

The build composes the auto-wired markdown adapter with a small in-example
sidebar-synthesizer adapter that turns `sidebars.js` categories into
synthesized parent nodes (per PRD-404-R6). See `docs/amendments-queue.md`
A11 for the upstream wiring proposal.

The `docusaurus.config.js` reproduces the canonical PRD-701-R9 shape so
downstream adopters who use the full Docusaurus CLI have a copy-pasteable
config.
